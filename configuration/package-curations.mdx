Package curations correct invalid or missing metadata for third-party dependencies across all scans. Unlike [`.ort.yml` curations](/configuration/ort-yml#curations), which are scoped to a single repository, the global `curations.yml` applies to every project analyzed with it.

## When to use curations

Use `curations.yml` to:

- Correct a broken or missing VCS URL, revision, or path.
- Fix incorrect binary or source artifact URLs.
- Set a `concluded_license` when the scanner findings don't reflect the actual license.
- Map unmappable declared license strings to valid SPDX expressions.
- Mark a package as metadata-only (e.g., a Maven BOM file) so it is skipped during scanning.
- Mark a package as modified (e.g., a fork) to flag it for additional review.
- Override the source code origins priority (`VCS` vs. `ARTIFACT`).
- Attach custom labels for use in policy rules, reporter templates, or ORT plugins.

<Tip>
If both `concluded_license` and `authors` are curated for a package, and `skipConcluded` is enabled in `config.yml`, ORT skips downloading and scanning that package entirely.
</Tip>

<Warning>
To correct license findings detected in a dependency's source code (scanner false positives), use [package configurations](/configuration/package-configurations) instead of curations.
</Warning>

## File location

Place `curations.yml` at `$ORT_CONFIG_DIR/curations.yml`. Alternatively, split curations across multiple files in `$ORT_CONFIG_DIR/curations/` — ORT reads all YAML files in that directory recursively.

Both the file and directory locations can also be configured as [`FilePackageCurationProviders`](https://github.com/oss-review-toolkit/ort/blob/main/plugins/package-curation-providers/file/src/main/kotlin/FilePackageCurationProvider.kt) in `config.yml`. ORT can also use [ClearlyDefined](https://clearlydefined.io/) as a curation source.

## Curation structure

A curation entry identifies a package by its ORT ID and supplies override values under `curations`. All fields in `curations` are optional — only specify what needs to be corrected.

```yaml
- id: "Maven:com.example.app:example:0.0.1"
  curations:
    comment: "An explanation why the curation is needed or the reasoning for a license conclusion."
    purl: "pkg:Maven/com.example.app/example@0.0.1?arch=arm64-v8a#src/main"
    authors:
    - "Name of one author"
    - "Name of another author"
    cpe: "cpe:2.3:a:example-org:example-package:0.0.1:*:*:*:*:*:*:*"
    concluded_license: "Apache-2.0"
    declared_license_mapping:
      "license a": "Apache-2.0"
    description: "Curated description."
    homepage_url: "http://example.com"
    binary_artifact:
      url: "http://example.com/binary.zip"
      hash:
        value: "ddce269a1e3d054cae349621c198dd52"
        algorithm: "MD5"
    source_artifact:
      url: "http://example.com/sources.zip"
      hash:
        value: "ddce269a1e3d054cae349621c198dd52"
        algorithm: "MD5"
    vcs:
      type: "Git"
      url: "http://example.com/repo.git"
      revision: "1234abc"
      path: "subdirectory"
    is_metadata_only: true
    is_modified: true
    source_code_origins: [ARTIFACT, VCS]
    labels:
      my-key: "my-value"
```

All available fields are defined in [`PackageCurationData.kt`](https://github.com/oss-review-toolkit/ort/blob/main/model/src/main/kotlin/PackageCurationData.kt).

### Package ID format

The `id` field follows ORT's coordinate format: `Type:Namespace:Name:Version`.

- Omitting the version applies the curation to **all versions** of the package.
- [Ivy-style version matchers](https://ant.apache.org/ivy/history/2.5.0/settings/version-matchers.html) are supported (e.g., `[0.21.0,0.25.0]` matches a range).

## Common examples

<AccordionGroup>
  <Accordion title="Fix a VCS URL (repository moved)">
    ```yaml
    - id: "Maven:asm:asm"
      curations:
        comment: "Repository moved to https://gitlab.ow2.org."
        vcs:
          type: "Git"
          url: "https://gitlab.ow2.org/asm/asm.git"
    ```

    No version is specified, so this curation applies to all versions of `asm:asm`.
  </Accordion>

  <Accordion title="Set a missing VCS revision">
    ```yaml
    - id: "NPM::ast-traverse:0.1.0"
      curations:
        comment: "Revision found by comparing the NPM package with the sources from https://github.com/olov/ast-traverse."
        vcs:
          revision: "f864d24ba07cde4b79f16999b1c99bfb240a441e"

    - id: "NPM::ast-traverse:0.1.1"
      curations:
        comment: "Revision found by comparing the NPM package with the sources from https://github.com/olov/ast-traverse."
        vcs:
          revision: "73f2b3c319af82fd8e490d40dd89a15951069b0d"
    ```
  </Accordion>

  <Accordion title="Set a concluded license for a version range">
    ```yaml
    - id: "NPM::ramda:[0.21.0,0.25.0]"
      curations:
        comment: >-
          The package is licensed under MIT per `LICENSE` and `dist/ramda.js`. The project logo
          is CC-BY-NC-SA-3.0 but it is not part of the distributed .tar.gz package.
        concluded_license: "MIT"
    ```
  </Accordion>

  <Accordion title="Mark a package as metadata-only (Maven BOM)">
    ```yaml
    - id: "Maven:org.jetbrains.kotlin:kotlin-bom"
      curations:
        comment: "The package is a Maven BOM file and thus is metadata only."
        is_metadata_only: true
    ```

    The downloader and scanner skip metadata-only packages. Evaluator rules can also opt to skip them.
  </Accordion>

  <Accordion title="Map an unmappable declared license string">
    ```yaml
    - id: "PyPI::pyramid-workflow:1.0.0"
      curations:
        comment: "The package has an unmappable declared license entry."
        declared_license_mapping:
          "BSD-derived (http://www.repoze.org/LICENSE.txt)": "LicenseRef-scancode-repoze"

    - id: "PyPI::branca"
      curations:
        comment: "A copyright statement was used to declare the license."
        declared_license_mapping:
          "Copyright (C) 2013, Martin Journois": "NONE"
    ```

    Multiple curations can each contribute mappings — they are merged together. The result appears in the `declared_license_processed` property of the curated package.
  </Accordion>

  <Accordion title="Override source code origin">
    ```yaml
    - id: "Maven:androidx.collection:collection:"
      curations:
        comment: "Scan the source artifact, because the VCS revision and path are hard to figure out."
        source_code_origins: [ARTIFACT]
    ```
  </Accordion>

  <Accordion title="Attach custom labels">
    ```yaml
    - id: "Maven:androidx.collection:collection:"
      curations:
        comment: "Specify the platform for use within policy rules."
        labels:
          platform: "android"

    - id: "NPM:@types:mime-types:2.1.0"
      curations:
        comment: "Retrieve vulnerabilities from Black Duck by the provided origin-id."
        labels:
          black-duck:origin-id: "npmjs:@types/mime-types/2.1.0"
    ```
  </Accordion>
</AccordionGroup>

## Command line usage

<Steps>
  <Step title="Place the curations file">
    Copy `curations.yml` to `$ORT_CONFIG_DIR/curations.yml`. The analyzer picks it up automatically.
  </Step>
  <Step title="Run the analyzer">
    ```bash
    cli/build/install/ort/bin/ort analyze \
      -i [source-code-dir] \
      -o [analyzer-output-dir]
    ```
  </Step>
  <Step title="Override curations during evaluation (optional)">
    Pass a curations file or directory directly to the evaluator for testing:

    ```bash
    cli/build/install/ort/bin/ort evaluate \
      -i [scanner-output-dir]/scan-result.yml \
      -o [evaluator-output-dir] \
      --license-classifications-file $ORT_CONFIG_DIR/license-classifications.yml \
      --package-curations-file $ORT_CONFIG_DIR/curations.yml \
      --package-curations-dir $ORT_CONFIG_DIR/curations \
      --rules-file $ORT_CONFIG_DIR/evaluator.rules.kts
    ```
  </Step>
</Steps>
