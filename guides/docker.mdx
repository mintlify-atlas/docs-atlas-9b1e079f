title: "Running ORT with Docker"
description: "How to pull, build, and run ORT using Docker, including volume mounts, credentials, and CI pipeline patterns."
---

ORT is distributed as a Docker image that bundles all required tools. This means you do not need to install Java, Python, Node.js, or any package manager toolchain locally — everything is included.

The official image is published at:

```
ghcr.io/oss-review-toolkit/ort
```

## Pulling the official image

```bash
docker pull ghcr.io/oss-review-toolkit/ort
docker run ghcr.io/oss-review-toolkit/ort --help
```

Two image variants are available:

- **`run` (default)** — includes all supported package managers (Python, Node.js, Ruby, Rust, Go, Scala, Swift, .NET, Android, and more).
- **`minimal`** — includes a smaller selection of the most common package managers.

## Running ORT commands

<Steps>
  <Step title="Mount your project directory">
    ORT needs to read your source code and write results to disk. Use `-v` to mount your working directory into the container:

    ```bash
    docker run \
      -v $PWD/:/project \
      ghcr.io/oss-review-toolkit/ort --info \
      analyze -i /project -o /project/ort-results
    ```

    The `/` between `$PWD` and `:` is required for PowerShell compatibility on Windows.
  </Step>

  <Step title="Provide an ORT config file">
    Mount an ORT configuration directory or file alongside your project:

    ```bash
    docker run \
      -v $PWD/:/project \
      ghcr.io/oss-review-toolkit/ort --info \
      -c /project/.ort/config.yml \
      analyze -i /project -o /project/ort-results
    ```

    ORT looks for its config in `~/.ort/config/` by default inside the container, which maps to `/home/ort/.ort/config/`.
  </Step>

  <Step title="Run the full pipeline">
    Chain analyze → scan → report by running separate `docker run` commands, passing results between stages:

    ```bash
    # Step 1: Analyze
    docker run \
      -v $PWD/:/project \
      ghcr.io/oss-review-toolkit/ort \
      analyze -i /project -o /project/ort-results/analyzer

    # Step 2: Scan
    docker run \
      -v $PWD/:/project \
      ghcr.io/oss-review-toolkit/ort \
      scan \
        -i /project/ort-results/analyzer/analyzer-result.yml \
        -o /project/ort-results/scanner

    # Step 3: Report
    docker run \
      -v $PWD/:/project \
      ghcr.io/oss-review-toolkit/ort \
      report \
        -f CycloneDX,SpdxDocument,StaticHTML \
        -i /project/ort-results/scanner/scan-result.yml \
        -o /project/ort-results/reporter
    ```
  </Step>
</Steps>

## Analyzing a subproject

To analyze only a subdirectory, adjust the `-i` path. You still need to mount the repository root so ORT can read VCS metadata:

```bash
docker run \
  -v $PWD/:/project \
  ghcr.io/oss-review-toolkit/ort \
  analyze -i /project/subproject -o /project/ort-results/analyzer
```

## Building the image locally

If you want to build a custom image — for example to pin tool versions or add internal certificates — clone the repository and build with Docker BuildKit:

```bash
git clone https://github.com/oss-review-toolkit/ort.git
cd ort
DOCKER_BUILDKIT=1 docker build -t ort .
```

Use the helper script to automatically set the ORT version from the current Git revision:

```bash
scripts/docker_build.sh
```

### Configuring JVM memory

The container image respects the `JAVA_OPTS` environment variable. Set it to increase the heap size for large projects:

```bash
docker run \
  -e JAVA_OPTS="-Xmx8g" \
  -v $PWD/:/project \
  ghcr.io/oss-review-toolkit/ort \
  analyze -i /project -o /project/ort-results/analyzer
```

## Authenticating with private repositories

ORT uses the standard `.netrc` file for Git credentials. To authenticate with a private repository:

<Steps>
  <Step title="Create a .netrc file">
    ```
    machine github.com login myusername password mypersonaltoken
    ```

    All properties must be on a single line. Newline-separated format is not supported by all tools included in the image.
  </Step>

  <Step title="Mount .netrc into the container">
    ```bash
    docker run \
      -v $PWD/:/project \
      -v $HOME/.netrc:/home/ort/.netrc \
      ghcr.io/oss-review-toolkit/ort \
      analyze -i /project -o /project/ort-results/analyzer
    ```

    <Warning>
      Make sure the `.netrc` file already exists at the path before running the command. If it does not exist, Docker will create a directory named `.netrc` instead of mounting a file.
    </Warning>
  </Step>
</Steps>

Alternatively, configure access via `.gitconfig`:

```bash
# .gitconfig
[url "https://myusername:mypersonaltoken@github.com"]
    insteadOf = https://github.com
```

```bash
docker run \
  -v $PWD/:/project \
  -v $HOME/.gitconfig:/home/ort/.gitconfig \
  ghcr.io/oss-review-toolkit/ort \
  analyze -i /project -o /project/ort-results/analyzer
```

## Installing custom certificates

If you are behind a corporate proxy or need to trust internal certificate authorities, pass custom certificates at image build time:

```bash
# Copy your .crt files into the build context first
cp /path/to/my-corp-ca.crt .

DOCKER_BUILDKIT=1 docker build \
  --build-arg CRT_FILES="my-corp-ca.crt" \
  -t ort-internal .
```

The certificates are imported into the Java keystore, the system trust store, and the Node.js trust store during the build.

### Fixing SSL handshake errors

Some corporate proxies do not support TLS 1.3. To force TLS 1.2, add this to your Dockerfile below the `FROM` line:

```dockerfile
ENV JAVA_OPTS="-Djdk.tls.client.protocols=TLSv1.2"
```

## Using ORT in CI pipelines

The Docker image is well-suited for CI use. See [CI integrations](/guides/ci-integrations) for ready-to-use configurations for GitHub Actions, GitLab CI, Jenkins, and Tekton.

A minimal GitHub Actions step looks like this:

```yaml
- name: Run ORT analyze
  run: |
    docker run \
      -v ${{ github.workspace }}:/project \
      ghcr.io/oss-review-toolkit/ort \
      analyze -i /project -o /project/ort-results/analyzer
```

<Tip>
  Cache the ORT data directory (`~/.ort`) between runs to speed up repeated scans. Mount it with `-v ort-data:/home/ort/.ort` using a named Docker volume, or persist it in your CI cache.
</Tip>
