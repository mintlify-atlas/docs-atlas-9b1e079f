---
title: "Package managers"
description: "ORT supports 30+ package managers across all major languages, detecting dependencies without modifying your source code."
---

The analyzer queries your project's package managers to build a complete dependency tree. No build system plugins or modifications to your source code are required — ORT reads the existing lock files and manifest files that your project already has.

The dependency tree is written to `analyzer-result.yml` in your chosen output directory and serves as the input for all downstream ORT tools.

<Note>
All projects must be buildable in a single step without custom environment configuration for analysis to succeed reliably.
</Note>

## Fallback package managers

When a project uses a build system that ORT does not directly support, or no package manager at all, two generic fallbacks are available.

<AccordionGroup>

<Accordion title="ORT Project file">

The ORT Project package manager lets you manually define projects and their dependencies in a hand-authored ORT Project definition file. This is useful for projects with unusual build systems or for describing a dependency graph that cannot be automatically detected.

The ORT Project package manager reads a hand-authored definition file (typically named `ort-project.ort.yml`) where you manually specify packages and their metadata. This is useful for projects with unusual build systems or for describing a dependency graph that cannot be automatically detected.

</Accordion>

<Accordion title="SPDX document file">

ORT can read [SPDX 2.x documents](https://spdx.dev/specifications/) as both project descriptors and package descriptors. This is useful when a package author already publishes an SPDX SBOM that describes their dependencies.

</Accordion>

</AccordionGroup>

## Enabling and disabling package managers

By default, ORT runs all supported package managers. You can restrict which ones are active in your ORT configuration file (`~/.ort/config/config.yml` or the path passed via `-c`).

<CodeGroup>

```yaml Enable specific package managers only
ort:
  analyzer:
    enabledPackageManagers: [Gradle, Maven, NPM]
```

```yaml Disable specific package managers
ort:
  analyzer:
    disabledPackageManagers: [Bower, Bundler]
```

</CodeGroup>

<Note>
`disabledPackageManagers` takes precedence over `enabledPackageManagers`. A package manager listed in both will be disabled.
</Note>

### Per-package-manager options

Some package managers accept additional options in the `packageManagers` section. For example:

```yaml
ort:
  analyzer:
    packageManagers:
      Gradle:
        mustRunAfter: [NPM]

      GradleInspector:
        options:
          javaHome: "/path/to/java/home"

      Yarn2:
        options:
          disableRegistryCertificateVerification: false

      Conan:
        options:
          lockfileName: "conan.lock"
          useConan2: true
```

The `mustRunAfter` option is useful when one package manager generates files that another package manager needs as input.

## Running the analyzer

```bash
ort analyze \
  -i /path/to/your/project \
  -o /path/to/output/dir
```

The result is written to `/path/to/output/dir/analyzer-result.yml`. Pass this file as `-i` to the scanner, advisor, evaluator, and reporter tools.
