title: "Analyzer"
description: "Determine the full dependency tree of your software projects without modifying source code."
---

The analyzer is ORT's Software Composition Analysis (SCA) engine. It interrogates the package managers present in your project directory to build a complete dependency graph — without applying plugins or touching your build configuration.

It is the **only mandatory ORT tool** to run. Every other tool (`scanner`, `advisor`, `evaluator`, `reporter`) consumes the `analyzer-result.yml` file it produces.

<Note>
The analyzer does not modify your source code or build files. It queries existing package manager metadata in place.
</Note>

## Prerequisites

- A checked-out project directory with at least one supported package manager definition file.
- All dependencies must be resolvable from the configured registries (i.e., the project must build cleanly).

## CLI usage

```bash
ort analyze -i <project-dir> -o <output-dir> [options]
```

| Flag | Short | Description |
|---|---|---|
| `--input-dir` | `-i` | Project directory to analyze. Can point to a directory or a single definition file (required). |
| `--output-dir` | `-o` | Directory where `analyzer-result.yml` is written (required). |
| `--output-formats` | `-f` | Comma-separated list of output formats: `YAML` (default) or `JSON`. |
| `--repository-configuration-file` | | Override the `.ort.yml` file found in the repository root. |
| `--resolutions-file` | | File containing issue resolutions (defaults to `~/.ort/config/resolutions.yml`). |
| `--label` | `-l` | Set a key=value label in the result. Repeatable. |
| `--dry-run` | | Show which package managers would be used without running the analysis. |

## Real example

```bash
ort analyze \
  -i /src/my-project \
  -o /results/analyzer \
  -f YAML,JSON \
  --label distribution=external \
  --label project=my-service
```

## Output format

The analyzer writes an `OrtResult` file named `analyzer-result.yml` (or `.json` if requested). The result embeds the full dependency graph for every detected project:

```yaml
analyzer:
  startTime: "2024-01-15T10:00:00Z"
  endTime: "2024-01-15T10:01:23Z"
  result:
    projects:
      - id: "Maven:com.example:my-service:1.0.0"
        definitionFilePath: "pom.xml"
        scopeDependencies:
          - name: "compile"
            dependencies:
              - id: "Maven:org.springframework:spring-core:6.1.0"
                dependencies: []
    packages:
      - metadata:
          id: "Maven:org.springframework:spring-core:6.1.0"
          declaredLicenses:
            - "Apache-2.0"
          homepageUrl: "https://spring.io/projects/spring-framework"
          sourceArtifact:
            url: "https://repo1.maven.org/maven2/..."
            hash:
              value: "abc123..."
              algorithm: "SHA-1"
```

## Configuration options

### `allowDynamicVersions`

By default, the analyzer rejects version ranges and dynamic version specifiers (e.g., `[1.0,)` in Maven, `^1.2.3` in NPM when they resolve differently from what is locked). Setting `allowDynamicVersions: true` in the ORT configuration relaxes this check:

```yaml
ort:
  analyzer:
    allowDynamicVersions: true
```

<Warning>
Enabling `allowDynamicVersions` may produce non-reproducible results. Prefer pinning versions or using a lockfile.
</Warning>

### Enabling and disabling package managers

You can restrict which package managers run via the ORT config file:

```yaml
ort:
  analyzer:
    enabledPackageManagers:
      - Gradle
      - Maven
    disabledPackageManagers:
      - NPM
```

## Scope exclusions via `.ort.yml`

Place a `.ort.yml` file at the repository root to exclude specific scopes (e.g., test dependencies) from analysis:

```yaml
excludes:
  scopes:
    - pattern: "test"
      reason: "TEST_DEPENDENCY_OF"
      comment: "Exclude test-only dependencies from compliance checks."
    - pattern: "devDependencies"
      reason: "DEV_DEPENDENCY_OF"
      comment: "Exclude development-only npm packages."
```

Excluded scopes are still present in the result file but are marked as excluded, so downstream tools can skip them.
